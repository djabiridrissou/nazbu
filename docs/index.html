// how it works

Three moves. Zero changes to your app.

01 · CAPTURE

Tap the change feed

Nazbu reads your database’s own stream of changes — Mongo change streams, Postgres WAL, MySQL binlog, SQLite sessions. Your app keeps writing to its DB exactly as before.

no app code changes
02 · REPLICATE

Stream over an encrypted swarm

Changes travel peer-to-peer. NAT hole-punching connects nodes on different networks directly across the internet — no shared wifi, no central server — with a relay fallback for the hardest NATs.

cross-network · e2e encrypted
03 · APPLY

Merge with real policies

Each peer writes into its own DB under per-collection rules: last-writer-wins, an append-only ledger for money that never merges lossily, or CRDT. Files travel coherently with the records that point at them.

financial-grade conflicts

// ship it

Plug in in five minutes.

One import, your existing connection, a room, and the rules for merging. That’s the whole integration.

server.js
import { Nazbu } from 'nazbu'

const nazbu = new Nazbu({
  db:    mongo,                      // your existing connection — nothing moves
  room:  `tenant-${tenantId}`,       // who this node syncs with
  policies: {
    ledgerEntries: 'append-only',     // money never merges lossily
    '*':           'last-writer-wins', // sensible default for the rest
  },
})

await nazbu.start()   // offline · multi-node · converging. that's it.

No new database. No server to operate. No schema migration. Delete Nazbu and your app still runs — it just stops surviving the outage.
